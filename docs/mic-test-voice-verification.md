# 마이크 테스트 음성 검증 로직 개선

`MicTestPanel`의 통과 기준이 너무 쉽게 뚫린다는 피드백을 받아 원인을 분석하고 개선한 과정, 검토했던 대안들, 채택하지 않은 이유를 정리한다.

## 1. 배경

기존 마이크 테스트(`src/components/mic-test-panel.tsx`)는 `AnalyserNode`로 읽은 평균 음량이 `VOICE_THRESHOLD`(0.08)를 200ms(`VOICE_SUSTAIN_MS`) 이상 끊기지 않고 넘으면 곧바로 통과 처리하는 구조였다. 실제 사용해보니 숨소리, 헛기침, 주변 잡음 등 발화가 아닌 소리로도 쉽게 통과된다는 지적이 있었다.

## 2. 문제의 본질: 볼륨 임계값만으로는 "사람 목소리인지"를 구분할 수 없다

임계값을 올리는 방안을 먼저 검토했으나, 이는 근본적인 해결이 아니었다. 볼륨 기반 검증은 "소리가 크다/작다"만 판단할 뿐, 그 소리가 사람 목소리인지 선풍기·에어컨 같은 주변 소음인지는 애초에 구분하지 못하는 방식이기 때문이다. 임계값을 아무리 올려도 그보다 큰 잡음이 있으면 똑같이 통과된다.

## 3. 검토한 해결 방안

- **`@ricky0123/vad-web`(Silero VAD, ONNX 기반) 도입**
  머신러닝 모델로 실제 음성 활동을 판별하는 라이브러리. jsDelivr CDN에서 실측한 결과 `bundle.min.js`(~67KB) + `onnxruntime-web` JS 글루(~47KB) + WASM 런타임 바이너리(`ort-wasm-simd-threaded.wasm`, ~10.7MB) + ONNX 모델(`silero_vad_v5.onnx`, ~2.2MB)로 **총 13MB 내외**가 필요했다. 이 프로젝트의 `package.json`은 UI 라이브러리 위주로 매우 가벼운데(ML/WASM 계열 의존성 없음), 마이크 테스트 하나를 위해 프로젝트 전체 번들보다 큰 에셋을 추가하는 건 특히 모바일 환경에서 부담이 크다고 판단해 채택하지 않았다. 정확도가 더 필요해지면 재검토 가능한 옵션으로 남겨둔다.
- **볼륨 임계값(`VOICE_THRESHOLD`)만 상향** — 위 2절의 이유로 근본적 해결이 아니라 기각.
- **스펙트럼 평탄도(spectral flatness) 게이트 추가 (채택)** — 아래 4절.

## 4. 채택안: 스펙트럼 평탄도 기반 게이트

`spectralFlatness()`(`src/components/mic-test-panel.tsx`)는 사람 목소리 대역(300~3400Hz, `SPEECH_BAND_HZ`)의 주파수 빈에서 기하평균/산술평균 비율을 계산한다. 사람의 유성음(모음 등)은 성대 진동에 의한 배음(harmonic) 구조로 에너지가 특정 주파수에 몰려 이 값이 낮게 나오고, 백색소음처럼 에너지가 고르게 퍼진 잡음은 이 값이 1에 가깝게 나온다. `avg >= VOICE_THRESHOLD`(음량 조건)와 `flatness <= SPECTRAL_FLATNESS_THRESHOLD`(0.5, 목소리 조건)를 모두 만족하는 상태가 200ms 이어져야 통과되도록 바꿨다.

### 왜 포먼트가 계속 움직여도 이 지표가 유효한가

발화 중 포먼트(formant) 위치는 음소가 바뀔 때마다 끊임없이 이동한다는 지적이 있었다. 그러나 스펙트럼 평탄도는 "에너지가 **어느** 주파수에 있는가"가 아니라 "에너지가 **몰려있는가, 퍼져있는가**"만 보는 지표라, 포먼트가 어디로 움직이든 유성음이라면 매 순간 국소적으로는 여전히 뾰족한(낮은 평탄도) 스펙트럼을 만든다. 포먼트 이동은 피크의 좌표를 바꿀 뿐 피크 자체를 없애지 않으므로, 이 지표를 무력화하지 않는다.

다만 실제로 신경 써야 할 약점은 따로 있었다.

- **무성음 구간**: /s/, /f/, /h/처럼 성대 진동 없는 난류성 소리는 물리적으로 잡음과 스펙트럼이 비슷해 평탄도가 높게 나온다. 이는 오탐이 아니라 정확한 판정이며, 200ms 연속 조건이라 문장 안에 섞인 유성음 구간에서 스트릭이 다시 쌓이므로 실사용에 지장이 없다고 판단했다.
- **주파수 해상도 부족**: 초기 구현은 `fftSize = 256`을 그대로 썼는데, 이 경우 빈 하나가 약 187Hz(48kHz 샘플레이트 기준)라 300~3400Hz 대역에 빈이 16개 정도만 들어간다. 저음 화자(F0 ~100Hz)는 배음 간격이 빈 폭보다 좁아 배음 구조가 뭉개져 평탄도 값이 부정확해질 수 있었다. 이는 포먼트 이동과 무관한 순수 FFT 해상도 문제라, `fftSize`를 **256 → 1024**로 올려 대역 내 빈 개수를 약 68개로 늘렸다. `fftSize`를 올리면 한 스냅샷이 대표하는 오디오 구간(시간 해상도)이 5.3ms → 21.3ms로 넓어지지만, 200ms 단위로 판정하는 이 용도에서는 무시할 수 있는 손실이라 트레이드오프상 유리하다고 판단했다.

### 안내 문구 세분화

기존에는 "볼륨 부족"과 "지속시간 부족" 두 가지만 구분했는데, "볼륨은 컸지만 목소리로 판정되지 않음"(잡음으로 추정) 케이스가 새로 생겨 `HINT_NOISE_DETECTED`를 추가했다. `everExceededThresholdRef`(음량 조건을 한 번이라도 넘었는지)와 `everVoiceLikeRef`(음량+목소리 조건을 한 번이라도 동시에 넘었는지)를 분리해서 셋 중 어느 문구를 보여줄지 판단한다.

## 5. 적용 범위: 마이크 테스트 화면에만 한정, 실제 답변 녹음은 영향 없음

이 변경은 `src/components/mic-test-panel.tsx`에만 적용했다. 실제 시험 답변 녹음(`src/features/exam/use-answer-recorder.ts`)은 완전히 별개의 훅으로, 여기도 `AnalyserNode`를 쓰지만 레벨 바 시각화 용도일 뿐 음성 검증 게이트가 없다. 녹음 자체는 `MediaRecorder`가 원본 오디오를 그대로 캡처해 업로드하는 구조라, `AnalyserNode`의 `fftSize`나 게이트 로직은 녹음 결과물에 영향을 주지 않는다. 두 훅을 공용화하지 않고 분리된 상태를 유지한 이유이기도 하다 — 마이크 테스트는 "통과/실패"를 판단해야 하지만 실제 녹음은 판단 없이 그대로 캡처만 하면 되는, 성격이 다른 용도이기 때문이다.

## 7. 후속 문제: "연속 200ms" 조건이 실제 발화 패턴보다 지나치게 엄격함

### 7.1 증상

실사용 중 볼륨과 목소리 판정(`isVoiceLike`)을 모두 통과하는 순간이 분명히 있는데도 "조금 더 길게 이야기해주세요"(`HINT_TOO_SHORT`) 안내가 계속 사라지지 않는 케이스가 보고됐다.

### 7.2 원인

기존 스트릭 로직은 `isVoiceLike`가 `false`인 프레임이 **단 한 번**만 나와도 `voiceStreakStartRef`를 즉시 `null`로 리셋했다.

```js
if (isVoiceLike) {
  // ...스트릭 누적...
} else {
  voiceStreakStartRef.current = null; // 단 한 프레임만 실패해도 완전 리셋
}
```

`requestAnimationFrame`은 약 16ms 주기로 도는데, `VOICE_SUSTAIN_MS`(200ms) 동안 끊김 없이 연속으로 `isVoiceLike`가 유지돼야 통과되는 구조였다. 그런데 실제 사람의 발화는:

- /s/, /f/, /h/, /ㅍ/, /ㅌ/, /ㅋ/ 같은 무성음 구간에서 스펙트럼이 순간적으로 평탄해져 `isVoiceLike`가 잠깐 `false`가 되고,
- 음절 사이, 숨쉬는 순간에도 아주 짧은 끊김이 자연스럽게 생긴다.

즉 "총 발화 시간"은 200ms를 훨씬 넘겨도 "완벽하게 끊김 없는 200ms"는 좀처럼 만들어지지 않아, 실제로는 정상적으로 말하고 있는데도 스트릭이 계속 0부터 다시 시작되는 문제였다.

### 7.3 검토한 해결 방안

- **A안: 짧은 끊김에 유예 시간을 두고 스트릭 리셋을 지연 (채택)** — 아래 7.4절.
- **B안: 롤링 윈도우 비율 판정** — "연속성" 요건 자체를 버리고, 최근 N ms(예: 1000ms) 동안의 프레임 중 voice-like 비율이 일정 % 이상이면 통과시키는 방식. 자연스러운 발화 패턴에 더 잘 맞고 근본적인 해결에 가깝다고 판단했으나, 링 버퍼 관리가 추가되고 `RATIO_WINDOW_MS`/`RATIO_THRESHOLD` 두 값을 새로 튜닝해야 해 변경 범위가 더 컸다. 우선 구현 리스크가 작은 A안을 적용하고, A안으로도 충분하지 않다는 게 확인되면 재검토하기로 했다.

### 7.4 채택안: 유예 시간(grace period)

`voiceStreakStartRef`를 매 프레임 실패마다 리셋하는 대신, 마지막으로 voice-like 판정을 받은 시각(`lastVoiceLikeAtRef`)을 기준으로 `VOICE_GAP_TOLERANCE_MS`(80ms) 이내의 끊김은 스트릭을 유지한 채 넘어가고, 그보다 긴 침묵만 진짜 끊김으로 간주해 리셋한다(`src/components/mic-test-panel.tsx`).

```js
if (isVoiceLike) {
  lastVoiceLikeAtRef.current = performance.now();
  if (voiceStreakStartRef.current === null) {
    voiceStreakStartRef.current = performance.now();
  } else if (
    performance.now() - voiceStreakStartRef.current >=
    VOICE_SUSTAIN_MS
  ) {
    verified = true;
  }
} else if (
  lastVoiceLikeAtRef.current === null ||
  performance.now() - lastVoiceLikeAtRef.current > VOICE_GAP_TOLERANCE_MS
) {
  voiceStreakStartRef.current = null;
}
```

기존 "연속 판정" 모델 자체는 그대로 두고 리셋 조건만 완화하는 최소 변경이라 구현 리스크가 작다. 다만 근본적으로 여전히 "거의 끊김 없는 200ms"를 요구하는 모델이라는 한계는 남아 있고, `VOICE_GAP_TOLERANCE_MS`(80ms) 값도 실측 없이 정한 초기값이다.

## 8. A안 폐기와 최종 채택안: 누적(비연속) 판정

### 8.1 A안이 실패한 이유

`VOICE_GAP_TOLERANCE_MS`를 80ms → 120ms로 올리고, 마이크 테스트 화면에 읽을 문장(`READING_PROMPT`, 7.6절이 아니라 이 시점에 추가)까지 보여줘서 발화 길이를 충분히 확보했는데도 실사용 테스트에서 스트릭이 184ms까지 쌓였다가 다시 0으로 끊기는 현상이 재현됐다. 화면에 임시로 추가한 디버그 표시(`avg`/`flatness`/streak 값 실시간 표시)로 확인해보니 `avg`와 `flatness`는 발화 중 대부분 조건을 만족하고 있었다 — 즉 문제는 음량이나 스펙트럼 판정이 아니라 여전히 "연속성" 그 자체였다.

원인은 A안의 구조적 한계였다. 그레이스 기간을 아무리 늘려도, A안은 결국 "한 번이라도 유예 시간을 넘는 침묵이 있으면 스트릭이 0부터 다시 시작"하는 모델이다. 그런데 실제 발화에는:

- 무성음 파열음(/p/, /t/, /k/ 등)의 완전 무음 폐쇄 구간
- 문장 내 쉼표·마침표에서의 자연스러운 pause
- 숨쉬기

가 반복적으로 섞여 있고, 이런 침묵 각각이 100~150ms를 넘는 경우가 드물지 않았다. 즉 문장을 길게 읽어도 "그 안에 끊김 없는 200ms 구간이 최소 한 번은 나온다"는 보장이 없었다 — 시행 횟수(문장 길이)를 늘려도 매 시행의 성공 확률 자체가 낮으면 결과는 크게 나아지지 않는다는 것이 실측으로 확인된 셈이다.

### 8.2 채택안: 누적 판정 (연속성 요건 제거)

"연속으로 끊기지 않아야 한다"는 요건을 아예 없애고, **목소리로 판정된(`isVoiceLike`) 시간의 총합**이 `VOICE_SUSTAIN_MS`(200ms)를 넘으면 통과시키는 방식으로 바꿨다(`src/components/mic-test-panel.tsx`). 이는 7.3절에서 검토했던 B안(롤링 윈도우 비율 판정)과 문제의식은 같지만, 윈도우·비율 두 값을 새로 튜닝해야 하는 B안보다 더 단순한 형태다 — 윈도우 없이 "녹음을 시작한 뒤로 지금까지 쌓인 총 발화 시간"만 본다.

```js
const now = performance.now();
const dt = lastTickAtRef.current === null ? 0 : now - lastTickAtRef.current;
lastTickAtRef.current = now;

if (isVoiceLike) {
  voiceAccumulatedMsRef.current += dt;
  if (voiceAccumulatedMsRef.current >= VOICE_SUSTAIN_MS) {
    verified = true;
  }
}
```

리셋 로직 자체가 없으므로 `voiceStreakStartRef`, `lastVoiceLikeAtRef`, `VOICE_GAP_TOLERANCE_MS`는 모두 제거했다. 프레임마다 `requestAnimationFrame`의 실제 경과 시간(`dt`)을 구해 누적하는 이유는, rAF 주기가 정확히 일정하지 않아 "프레임 수 × 고정 간격"으로 계산하면 실제 경과 시간과 오차가 누적될 수 있기 때문이다.

이 방식은 끊김이 몇 번 있든, 얼마나 길든 전혀 페널티가 없다. 대신 통과 기준이 "총 200ms 이상 목소리로 판정된 적이 있는가"로 완전히 느슨해지므로, 노이즈를 걸러내는 역할은 이제 전적으로 6절의 음량(`VOICE_THRESHOLD`)·스펙트럼 평탄도(`SPECTRAL_FLATNESS_THRESHOLD`) 게이트에 달려 있다. 다만 두 게이트가 이미 "볼륨은 큰데 목소리가 아님"(노이즈)과 "목소리가 아예 없음"을 구분하고 있어서, 연속성 요건을 없앤다고 해서 오탐(false positive)이 새로 생기는 건 아니라고 판단했다.

### 8.3 함께 추가한 보조 장치: 읽을 문장(`READING_PROMPT`)

마이크 테스트 화면에 토익 스피킹 실제 문제와 비슷한 톤의 영어 안내문(`READING_PROMPT`, 예: "Thank you for calling Riverside Electronics. ...")을 보여주고 소리 내어 읽게 했다. "아무 말이나 해보세요"보다 사용자가 실제로 몇 초간 이어 말하게 유도하는 효과가 있고, 토익 스피킹 앱이라는 맥락과도 자연스럽게 맞아 함께 채택했다. 다만 8.1절에서 확인했듯 이것만으로는 A안의 구조적 문제를 해결하지 못했고, 8.2절의 판정 방식 전환이 실제 해결책이었다.

### 8.4 남은 과제

- `VOICE_SUSTAIN_MS`(200ms)는 연속성 요건이 사라지면서 사실상 문턱이 더 낮아진 셈이라, 너무 쉽게 통과되지 않는지 재검토가 필요할 수 있다(예: 300~500ms로 상향).
- 마이크 테스트 화면에 남겨둔 실시간 디버그 표시(`avg`/`flatness`/누적 ms)는 진단용 임시 코드이므로, 임계값들이 실사용 기준으로 충분히 검증되면 제거한다.

## 9. 실측 기반 재조정: `SPECTRAL_FLATNESS_THRESHOLD` 0.8 → 0.97

8절까지 적용한 뒤에도 실사용 테스트에서 "주변 소음이 감지돼요"가 계속 뜨는 문제가 재현됐다. 디버그 표시로 실제 값을 확인해보니 `avg`는 0.124로 임계값(0.08)을 여유 있게 넘었지만, `flatness`는 또렷하게 말하는 상황에서도 0.957 정도로 임계값(0.8)을 한참 웃돌았다.

`getUserMedia`에서 `autoGainControl: false`로 마이크 자동 증폭을 꺼둔 구성상 캡처되는 신호 자체가 약해서, 목소리 피크와 배경 잡음 바닥의 상대적 대비가 작고, 그 결과 실제 목소리를 읽어도 스펙트럼 평탄도가 백색소음에 가까운 값으로 나온다는 게 실측으로 확인된 것이다. 이 구성에서 사람 목소리의 실측 평탄도가 0.95~0.96대라는 근거가 생겼으므로, 임계값을 `0.8` → `0.97`로 올려 실제 목소리를 잡음으로 오판하지 않도록 했다. 노이즈 판별 여력이 줄어드는 트레이드오프는 있지만, 애초에 이 값을 실측 없이 잡았던(6절 "남은 과제") 초기값이었던 만큼 실측치를 반영해 재조정한 것이다.

동시에 마이크 테스트에서 읽는 문장(`READING_PROMPT`)도 숫자·약어("a.m.", "p.m.")·쉼표가 많아 읽다가 자주 끊기던 것을, 쉬운 단어 위주의 두 문장("Welcome to our store. We hope you enjoy your shopping experience today.")으로 바꿔 막힘 없이 읽기 쉽게 했다.

## 10. 마이크 테스트 전용으로 AGC·노이즈 억제 활성화

9절 이후에도 인식률이 낮다는 피드백이 이어져, 비용이 가장 작은 가설부터 실험했다: `getUserMedia`에서 꺼두었던 브라우저 오디오 전처리를 마이크 테스트에서만 켜는 것.

**AGC(Automatic Gain Control, 자동 이득 제어)란?** 브라우저가 마이크 입력의 크기를 실시간으로 감시하면서 증폭률을 자동으로 조절해주는 기능이다. 작게 말하면 증폭을 키우고 크게 말하면 줄여서, 목소리가 항상 일정한 레벨 근처로 들어오게 만든다. 화상회의 앱에서 "마이크에서 멀어져도 목소리 크기가 비슷하게 들리는" 게 바로 이것. 노트북 내장 마이크의 원시 신호는 보통 꽤 약한데, AGC를 꺼두면 그 약한 신호가 그대로 들어온다 — 9절에서 확인한 "말해도 avg가 0.12 정도"였던 이유다.

**noiseSuppression**은 에어컨·팬처럼 일정하게 지속되는 배경 소음의 패턴을 학습해서 빼주는 필터다. 함께 켜면 "잡음이 목소리로 오판될" 여지가 줄어든다. **echoCancellation**은 스피커로 재생 중인 소리가 마이크로 되돌아오는 것을 제거하는 기능인데, 마이크 테스트 중에는 재생하는 소리가 없어 그대로 꺼두었다.

이 변경이 안전한 이유는 5절과 같다 — 실제 답변 녹음(`use-answer-recorder.ts`)은 자체 `getUserMedia` 설정을 쓰는 완전히 별개의 훅이라, 채점용 음성은 여전히 전처리 없이 원본 그대로 캡처된다.

결과: `avg`(음량)는 확실히 개선됐지만, **평탄도는 거의 그대로였다**(발화 중 p50 0.957 — 9절의 AGC 끈 구성과 사실상 동일). 즉 신호 레벨은 문제의 일부일 뿐이었고, 진짜 원인은 다른 곳에 있었다. → 11절.

## 11. 진짜 근본 원인: 평탄도를 "로그 스케일 위에서" 계산하고 있었다

### 11.1 실측으로 원인을 좁힌 과정

10절 실험과 함께 디버그 표시를 강화했다. 순간값은 눈으로 쫓기 어려워서(100ms마다 갱신), 세션 통계를 함께 보여주도록 했다:

- **p50**: 음량 조건을 넘긴 프레임들의 평탄도 **중앙값** — "말하는 동안의 평소 평탄도"
- **min**: 같은 프레임들의 평탄도 최소값
- **voice / loud**: 두 조건(음량+평탄도)을 모두 만족해 실제로 쌓인 시간 vs 음량 조건만 만족한 시간 — 통과가 안 될 때 음량 탓인지 평탄도 탓인지 즉시 구분된다

이 표시로 결정적인 모순이 드러났다: 발화 중 평탄도가 **최소 0.315까지 내려가는데 중앙값은 0.957**. 순간적으로는 목소리다운 값이 찍히지만 평소에는 백색소음 수준이라는 뜻이다. 최소값(0.2대)을 보고 임계값을 0.6으로 조였더니 문장을 다 읽어도 통과가 안 되는 문제가 바로 재현됐다 — 유성음 프레임 대부분이 0.6을 넘고 있었던 것. 그리고 AGC를 켜도 p50이 전혀 움직이지 않았다는 사실이 "레벨 문제가 아니라 **계산 자체의 문제**"라는 결정적 단서가 됐다.

### 11.2 원인: dB(로그) 값을 선형 값인 것처럼 평균냈다

`getByteFrequencyData()`가 돌려주는 0~~255는 소리 에너지 그 자체가 아니라, **데시벨(dB)을 0~~255 구간에 눌러 담은 값**이다(기본 설정: -100dB → 0, -30dB → 255).

dB는 로그 단위다. 그런데 스펙트럼 평탄도는 "배음 피크가 골짜기보다 에너지가 몇 배 큰가"를 재는 지표다(기하평균/산술평균 — 값들이 서로 비슷하면 1, 일부만 툭 튀면 0에 가까워진다). 구체적인 숫자로 보면:

- 목소리의 배음 피크가 -40dB, 배음 사이 골짜기가 -70dB라고 하자. **에너지로는 1000배 차이**다.
- 이걸 바이트로 바꾸면 피크 ≈ 218, 골 ≈ 109. **겨우 2배 차이**로 압축된다.

값들이 2배 안쪽에서 고만고만하니 기하평균 ≈ 산술평균, 즉 평탄도가 항상 1 근처로 나온다. **진짜 목소리를 넣어도 0.95+가 나왔던 이유, 그리고 임계값이 0.5 → 0.8 → 0.97로 계속 밀려 올라갔던 이유가 전부 이것이다.** 처음부터 지표가 변별력을 낼 수 없는 도메인에서 계산되고 있었고, 우리는 그동안 임계값 조정으로 증상만 보정해온 셈이다. (6절의 "0.5는 실측 없이 정한 초기값" — 그 0.5는 선형 도메인 기준의 교과서적 값이었는데, 실제 계산은 로그 도메인이었으니 맞을 수가 없었다.)

가끔 0.3대 최소값이 찍혔던 것은 발화 시작 순간처럼 일부 빈만 에너지를 갖고 나머지가 바닥(-100dB, 바이트 0)에 깔리는 프레임에서만 로그 압축을 뚫고 차이가 드러났기 때문이다.

### 11.3 해결: 바이트를 선형 파워로 되돌린 뒤 계산

`spectralFlatness()`에서 각 바이트를 원래의 선형 파워로 복원한 뒤 평탄도를 계산하도록 바꿨다:

```
바이트 v → dB = -100 + v × (70/255) → 파워 = 10^(dB/10)
```

코드에서는 `10^(v × 70/255/10)`만 계산한다 — 상수 오프셋(-100dB)은 모든 빈에 같은 배율(10^-10)로 곱해질 뿐이라 기하평균/산술평균의 **비율**에서는 소거되기 때문이다(`BYTE_TO_POWER_EXP` 상수).

결과는 실측으로 확인됐다: **발화 중 p50은 0.5보다 확실히 아래, 무음·배경 소음은 1 근처** — 드디어 목소리와 잡음이 다른 값 대역으로 갈라졌다. 임계값은 `0.5`로 재설정했다(선형 도메인이라 이전 값들과 스케일 자체가 다르다).

### 11.4 교훈

- **어떤 지표를 계산하기 전에, 입력이 선형인지 로그인지부터 확인하라.** Web Audio의 `getByteFrequencyData`/`getFloatFrequencyData`는 둘 다 dB 스케일이다. 평탄도처럼 "배율"을 보는 지표는 선형 도메인으로 되돌린 뒤 계산해야 한다.
- **임계값이 이론적 극단(여기서는 1.0) 바로 앞까지 밀린다면, 튜닝 문제가 아니라 지표 자체를 의심하라.** 0.97이라는 임계값은 사실상 "이 게이트는 아무것도 거르지 못한다"는 신호였다.
- **순간값 디버깅이 안 되면 분포(중앙값·최소값)와 분해 카운터(voice/loud)를 보여줘라.** p50 하나가 원인 특정을 가능하게 했다.

## 12. UT 피드백: 문장을 다 읽기도 전에 통과되는 문제 → STT 대신 "판단 주체를 사용자에게 이전"

### 12.1 문제

사용자 테스트(UT) 중 "문장을 다 읽지도 않았는데 마이크 테스트가 끝나버린다"는 피드백이 나왔다. 원인은 8절에서 채택한 누적 판정 자체의 구조에 있었다 — 목소리로 판정된 시간의 총합이 `VOICE_SUSTAIN_MS`(당시 400ms)를 넘는 순간 곧바로 다음 화면(사운드 체크)으로 자동 전환됐기 때문에, `READING_PROMPT` 문장을 끝까지 읽기 전이라도(문장 초반 몇 단어만으로 400ms는 금방 채워진다) 화면이 넘어가버렸다.

### 12.2 검토한 방안: STT로 실제 발화 내용까지 검증 (기각)

가장 정확한 해결책은 STT(Speech-to-Text)를 붙여 사용자가 실제로 `READING_PROMPT` 문장을 발화했는지 텍스트 매칭까지 검증하는 것이다. 하지만 이 화면의 목적에 비해 과한 설계라고 판단해 채택하지 않았다.

- 마이크 테스트는 "마이크가 목소리를 정상적으로 캡처하는가"만 확인하면 되는 화면이다. 발화 **내용**이 문장과 일치하는지까지 검증할 이유가 없다 — 실제 시험 답변의 내용 채점은 별도 백엔드 AI가 담당한다.
- STT 엔진(브라우저 Web Speech API든 서버 API든)을 추가하면 응답 지연, 인식 실패 처리, 발음/억양 오차 허용 범위 같은 새로운 복잡도가 딸려온다. 3절에서 VAD 라이브러리(~13MB)를 "이 화면의 목적에 비해 비용이 과하다"는 이유로 기각한 것과 같은 판단 기준이다.
- 근본 원인은 "판정이 부정확하다"가 아니라 "판정 로직이 사용자 대신 알아서 화면을 끝내버린다"는 것이었다. 필요한 건 더 정교한 자동 판정이 아니라, 애초에 자동으로 끝내지 않는 것이었다.

### 12.3 채택안: 판단 주체를 사용자에게 이전 (수동 종료 + 재생 검토 + 지속시간 게이트 제거)

`src/components/mic-test-panel.tsx`에서 두 가지를 함께 바꿨다.

1. **자동 전환 제거, 수동 종료 + 재생 검토 추가**: 목소리 판정이 임계값을 넘어도 더 이상 화면을 자동으로 전환하지 않는다. 사용자가 문장을 끝까지 읽고 직접 "다 읽었어요" 버튼을 눌러야 녹음이 끝난다(`idle → recording → reviewing` 단계). 이 시점에 `MediaRecorder`로 실제 오디오를 녹음해뒀다가(기존에는 `AnalyserNode`로 실시간 분석만 하고 오디오 자체는 저장하지 않았다) 방금 녹음한 목소리를 재생해서 다시 들어볼 수 있는 화면을 보여준다.
2. **지속시간 게이트(`VOICE_SUSTAIN_MS`) 제거**: 이제 "언제 끝낼지"를 사용자가 정하므로, 통과 조건도 "녹음 중 목소리로 판정된 순간이 한 번이라도 있었는가"(`everVoiceLikeRef`)로 단순화했다. 몇 ms를 채워야 통과되는지를 자동으로 재는 대신, "충분히 길게/또렷하게 말했는지"는 재생 검토 화면에서 사용자 본인이 자기 목소리를 들어보고 판단하게 한다. 이에 따라 누적 시간 계산(`voiceAccumulatedMsRef`, `lastTickAtRef`, `dt` 누적)과 "조금 더 길게 이야기해주세요"(`HINT_TOO_SHORT`) 안내 문구도 함께 제거했다.

음량(`VOICE_THRESHOLD`)·스펙트럼 평탄도(`SPECTRAL_FLATNESS_THRESHOLD`) 게이트는 그대로 남겨뒀다. 이 둘은 사용자가 스스로 눈치채기 어려운 문제(마이크가 꺼져 있거나, 파형은 움직이는데 실제로는 잡음만 잡히는 경우)를 걸러내는 역할이라 자동 판정으로 유지할 가치가 있다고 봤다. 반면 "얼마나 길게 말했는가"는 녹음하면서, 그리고 재생 화면에서 본인이 직접 확인할 수 있는 정보라 자동 게이트로 이중 체크할 필요가 없다고 판단했다.

### 12.4 남은 과제

- 재생 검토 화면에서 재생 버튼을 한 번도 누르지 않고 바로 "다음"을 눌러도 다음 단계로 넘어갈 수 있다 — 재생을 강제하지는 않는다. 강제해야 한다는 피드백이 나오면 재검토한다.
