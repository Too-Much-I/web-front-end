# 마이크 테스트 음성 검증 로직 개선

`MicTestPanel`의 통과 기준이 너무 쉽게 뚫린다는 피드백을 받아 원인을 분석하고 개선한 과정, 검토했던 대안들, 채택하지 않은 이유를 정리한다.

## 1. 배경

기존 마이크 테스트(`src/components/mic-test-panel.tsx`)는 `AnalyserNode`로 읽은 평균 음량이 `VOICE_THRESHOLD`(0.08)를 200ms(`VOICE_SUSTAIN_MS`) 이상 끊기지 않고 넘으면 곧바로 통과 처리하는 구조였다. 실제 사용해보니 숨소리, 헛기침, 주변 잡음 등 발화가 아닌 소리로도 쉽게 통과된다는 지적이 있었다.

## 2. 문제의 본질: 볼륨 임계값만으로는 "사람 목소리인지"를 구분할 수 없다

임계값을 올리는 방안을 먼저 검토했으나, 이는 근본적인 해결이 아니었다. 볼륨 기반 검증은 "소리가 크다/작다"만 판단할 뿐, 그 소리가 사람 목소리인지 선풍기·에어컨 같은 주변 소음인지는 애초에 구분하지 못하는 방식이기 때문이다. 임계값을 아무리 올려도 그보다 큰 잡음이 있으면 똑같이 통과된다.

## 3. 검토한 해결 방안

- **`@ricky0123/vad-web`(Silero VAD, ONNX 기반) 도입**
  머신러닝 모델로 실제 음성 활동을 판별하는 라이브러리. jsDelivr CDN에서 실측한 결과 `bundle.min.js`(~67KB) + `onnxruntime-web` JS 글루(~47KB) + WASM 런타임 바이너리(`ort-wasm-simd-threaded.wasm`, ~10.7MB) + ONNX 모델(`silero_vad_v5.onnx`, ~2.2MB)로 **총 13MB 내외**가 필요했다. 이 프로젝트의 `package.json`은 UI 라이브러리 위주로 매우 가벼운데(ML/WASM 계열 의존성 없음), 마이크 테스트 하나를 위해 프로젝트 전체 번들보다 큰 에셋을 추가하는 건 특히 모바일 환경에서 부담이 크다고 판단해 채택하지 않았다. 정확도가 더 필요해지면 재검토 가능한 옵션으로 남겨둔다.
- **볼륨 임계값(`VOICE_THRESHOLD`)만 상향** — 위 2절의 이유로 근본적 해결이 아니라 기각.
- **스펙트럼 평탄도(spectral flatness) 게이트 추가 (채택)** — 아래 4절.

## 4. 채택안: 스펙트럼 평탄도 기반 게이트

`spectralFlatness()`(`src/components/mic-test-panel.tsx`)는 사람 목소리 대역(300~3400Hz, `SPEECH_BAND_HZ`)의 주파수 빈에서 기하평균/산술평균 비율을 계산한다. 사람의 유성음(모음 등)은 성대 진동에 의한 배음(harmonic) 구조로 에너지가 특정 주파수에 몰려 이 값이 낮게 나오고, 백색소음처럼 에너지가 고르게 퍼진 잡음은 이 값이 1에 가깝게 나온다. `avg >= VOICE_THRESHOLD`(음량 조건)와 `flatness <= SPECTRAL_FLATNESS_THRESHOLD`(0.5, 목소리 조건)를 모두 만족하는 상태가 200ms 이어져야 통과되도록 바꿨다.

### 왜 포먼트가 계속 움직여도 이 지표가 유효한가

발화 중 포먼트(formant) 위치는 음소가 바뀔 때마다 끊임없이 이동한다는 지적이 있었다. 그러나 스펙트럼 평탄도는 "에너지가 **어느** 주파수에 있는가"가 아니라 "에너지가 **몰려있는가, 퍼져있는가**"만 보는 지표라, 포먼트가 어디로 움직이든 유성음이라면 매 순간 국소적으로는 여전히 뾰족한(낮은 평탄도) 스펙트럼을 만든다. 포먼트 이동은 피크의 좌표를 바꿀 뿐 피크 자체를 없애지 않으므로, 이 지표를 무력화하지 않는다.

다만 실제로 신경 써야 할 약점은 따로 있었다.

- **무성음 구간**: /s/, /f/, /h/처럼 성대 진동 없는 난류성 소리는 물리적으로 잡음과 스펙트럼이 비슷해 평탄도가 높게 나온다. 이는 오탐이 아니라 정확한 판정이며, 200ms 연속 조건이라 문장 안에 섞인 유성음 구간에서 스트릭이 다시 쌓이므로 실사용에 지장이 없다고 판단했다.
- **주파수 해상도 부족**: 초기 구현은 `fftSize = 256`을 그대로 썼는데, 이 경우 빈 하나가 약 187Hz(48kHz 샘플레이트 기준)라 300~3400Hz 대역에 빈이 16개 정도만 들어간다. 저음 화자(F0 ~100Hz)는 배음 간격이 빈 폭보다 좁아 배음 구조가 뭉개져 평탄도 값이 부정확해질 수 있었다. 이는 포먼트 이동과 무관한 순수 FFT 해상도 문제라, `fftSize`를 **256 → 1024**로 올려 대역 내 빈 개수를 약 68개로 늘렸다. `fftSize`를 올리면 한 스냅샷이 대표하는 오디오 구간(시간 해상도)이 5.3ms → 21.3ms로 넓어지지만, 200ms 단위로 판정하는 이 용도에서는 무시할 수 있는 손실이라 트레이드오프상 유리하다고 판단했다.

### 안내 문구 세분화

기존에는 "볼륨 부족"과 "지속시간 부족" 두 가지만 구분했는데, "볼륨은 컸지만 목소리로 판정되지 않음"(잡음으로 추정) 케이스가 새로 생겨 `HINT_NOISE_DETECTED`를 추가했다. `everExceededThresholdRef`(음량 조건을 한 번이라도 넘었는지)와 `everVoiceLikeRef`(음량+목소리 조건을 한 번이라도 동시에 넘었는지)를 분리해서 셋 중 어느 문구를 보여줄지 판단한다.

## 5. 적용 범위: 마이크 테스트 화면에만 한정, 실제 답변 녹음은 영향 없음

이 변경은 `src/components/mic-test-panel.tsx`에만 적용했다. 실제 시험 답변 녹음(`src/features/exam/use-answer-recorder.ts`)은 완전히 별개의 훅으로, 여기도 `AnalyserNode`를 쓰지만 레벨 바 시각화 용도일 뿐 음성 검증 게이트가 없다. 녹음 자체는 `MediaRecorder`가 원본 오디오를 그대로 캡처해 업로드하는 구조라, `AnalyserNode`의 `fftSize`나 게이트 로직은 녹음 결과물에 영향을 주지 않는다. 두 훅을 공용화하지 않고 분리된 상태를 유지한 이유이기도 하다 — 마이크 테스트는 "통과/실패"를 판단해야 하지만 실제 녹음은 판단 없이 그대로 캡처만 하면 되는, 성격이 다른 용도이기 때문이다.

## 6. 남은 과제

- `SPECTRAL_FLATNESS_THRESHOLD`(0.5)와 `VOICE_SUSTAIN_MS`(200ms)는 실측 튜닝 없이 정한 초기값이라, 실사용자 테스트(다양한 화자의 음높이, 마이크 기종, 배경 소음 종류)를 거쳐 조정이 필요할 수 있다.
- 지금은 이 두 값과 `VOICE_THRESHOLD`가 서로 다른 상수로 흩어져 있는데, 추후 잡음 유형별 오탐/오거부 사례가 쌓이면 재조정 기준을 마련해야 한다.
- 정확도를 더 높여야 한다면 3절에서 기각한 VAD 라이브러리 도입을 다시 검토할 수 있다(다만 13MB 에셋 비용을 감수할 만한 근거가 필요하다).
